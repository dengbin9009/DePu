# DePu 项目宪章

## 核心原则

### I. 规则正确性优先
所有牌局推进、合法动作、牌型评估、主池/边池拆分和结算逻辑必须以后端规则引擎为权威来源。前端不得自行推断会影响结果的规则，只能展示后端返回的状态、合法动作和错误。

### II. 长牌与短牌规则集可配置
长牌和短牌必须通过明确的规则集建模，包括牌组点数、顺子规则、牌型排序、盲注/前注结构和摊牌比较规则。新增玩法应增加规则集配置和测试，而不是把条件分散在界面或 API 控制器中。

### III. 测试优先覆盖高风险规则
实现任何规则变化前，必须先补充能失败的测试用例。最低覆盖范围包括：最佳五张牌、A 低顺子、短牌 A-6-7-8-9、短牌牌型排序、最小加注、全下、不足额加注、主池/边池、平分底池和非法动作不改变状态。

### IV. 版本边界必须显式声明
DePu 允许存在多个以 Spec Kit 文档定义的版本基线，但每个版本都必须显式声明自身边界，且不得无说明地混用能力范围。`001-poker-simulator` 是本地单机规则模拟器基线，不实现登录、真钱筹码、支付、多房间、多人实时同步、排行榜或反作弊；`002-multiplayer-poker-v1` 是多人对战产品基线，可以在保留规则正确性和测试优先原则的前提下引入账号、虚拟金币、房间和多人轮流操作。任何超出当前 feature 规格的能力，必须写入新的 feature 规格或未来扩展，而不是隐式并入现有版本。

### V. 可复盘、可解释、可恢复
每个改变牌局状态的动作都必须记录为行动日志，并能从保存的快照和日志恢复到可展示状态。错误响应必须说明失败原因和关联字段，不能静默修正用户输入或伪造成功。

## 技术约束

后端使用 Go，前端使用 Vue。`001-poker-simulator` 基线持久化使用 SQLite；`002-multiplayer-poker-v1` 基线生产默认使用 MySQL，开发/测试允许 SQLite。前后端通过 HTTP JSON API 通信。每个 feature 的 API 契约、数据模型和任务拆分必须与其对应的 Spec Kit 文档保持一致，不得跨 feature 隐式漂移。

## 开发流程

需求先进入 `spec.md`，技术决策进入 `plan.md` 和 `research.md`，实体进入 `data-model.md`，接口进入 `contracts/openapi.yaml`，执行拆分进入 `tasks.md`。完成实现前必须运行后端规则测试、API 集成测试和前端基础验证。

## 治理

本宪章优先于临时实现偏好。违反规则正确性、测试优先或已声明版本边界的变更，必须在计划中明确说明原因、风险和替代方案；若变更的是版本边界本身，必须同步更新宪章或建立新的 feature 基线说明。
