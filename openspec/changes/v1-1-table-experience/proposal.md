# 变更提案：V1.1 牌桌实时体验增强

## 背景

当前多人德州扑克 v1 已完成账号、钱包、建房、邀请码加入、正式多人轮流操作、每手牌结果保存与展示，并已将正式多人房间的实时同步、开局和玩家动作从 HTTP 轮询升级为 socket command 与事件广播。

V1.1 的目标是在 socket 基础上补齐更像“真实多人牌桌”的核心体验：玩家需要知道自己还剩多久操作，断线后能回到可信状态，房间内发生过什么动作可以被追踪，已结束手牌可以完整复盘，房间内输赢情况可以快速查看，并支持轻量聊天与表情互动。

## 目标

- 增加服务端权威的行动倒计时，并在超时后自动执行可预测动作。
- 增强断线重连体验，展示房间成员在线/离线状态，并在重连后恢复权威快照。
- 增加房间内实时动作日志，让玩家能看到入座、开局、行动、超时和结算过程。
- 增加正式多人手牌完整回放，支持已结算手牌按步骤复盘。
- 增加房间战绩榜，展示房间内玩家手数、胜场、最大赢池和净输赢。
- 增加房间聊天与预设表情，通过 socket 实时发送给房间成员。

## 非目标

- 不接入真实支付、退款、支付回调或运营后台。
- 不实现大厅匹配、好友、公会、跨房间排行榜或赛季系统。
- 不实现机器人补位、AI 代打或复杂托管策略。
- 不实现私聊、语音、图片、富文本或跨房间聊天。
- 不暴露任何会影响正式牌局公平性的前端规则计算。
- 不把独立规则测试页的调试设牌和测试回放能力并入正式多人主流程。

## 功能点

### 1. 行动倒计时与超时处理

服务端在每次轮到玩家行动时生成权威 `actionDeadlineAt`，通过 `hand.updated` 或计时事件同步给房间成员。前端只展示倒计时，不负责判定超时。

到达截止时间后，服务端自动提交超时动作：若当前局面允许 `check`，则自动 `check`；否则自动 `fold`。超时动作必须像普通动作一样进入规则引擎、持久化、广播并写入动作日志。

### 2. 断线重连与在线状态

socket 连接断开后，服务端标记该用户在房间内离线并广播状态。用户重新连接并订阅房间后，服务端返回包含房间、当前手牌、行动截止时间、动作日志摘要、聊天摘要和战绩榜摘要的最新快照。

断线不自动替玩家行动；若轮到该玩家且倒计时结束，则按统一超时规则处理。

### 3. 实时动作日志

正式房间必须展示可追溯的动作日志，覆盖入座、离座、开局、各街动作、超时自动动作、摊牌、结算和钱包结果提示。日志只能包含房间成员在当前时点允许看到的公开信息，不得提前泄露未展示手牌。

### 4. 完整手牌回放

已结算的正式多人手牌必须可以从房间历史进入回放。回放按服务端保存的公开步骤还原牌局过程，支持逐步查看行动、公共牌变化、底池变化和最终结算。未公开的底牌在回放中仍不得泄露。

### 5. 房间战绩榜

房间成员可以查看该房间内玩家的聚合表现，包括至少：参与手数、胜利手数、净输赢、最大赢得底池和最近结算时间。战绩榜以数据库中正式手牌结果为权威来源，并在新手牌结算后刷新或广播更新。

### 6. 聊天与表情

房间成员可以发送短文本聊天和预设表情。服务端必须校验发送者是房间成员、限制消息长度、限制频率，并通过 socket 向房间成员广播。聊天不参与牌局规则，也不得改变手牌状态。

## 用户价值

- 玩家能明确看到当前行动压力，牌局不会因为某个玩家不操作而卡死。
- 断线后能重新回到同一房间，不需要依赖手动刷新猜测当前状态。
- 房间成员能通过日志理解刚刚发生了什么，降低多人操作时的迷惑感。
- 已结束手牌可以复盘，方便解释输赢和验证结算。
- 房间内输赢表现可见，增强持续对局的反馈。
- 聊天和表情让邀请码房间具备基本社交氛围。

## 风险

- 倒计时必须以服务端时间为准，避免前端本地时间差导致误判。
- 超时动作会改变牌局状态，必须复用现有房间串行 command 处理，不能绕过规则引擎。
- 回放必须严格区分公开信息和未公开信息，避免泄露弃牌玩家底牌。
- 聊天需要最小限度的长度和频率限制，避免刷屏影响牌桌体验。
- 战绩榜、日志、回放、聊天都增加持久化或快照数据，必须控制查询范围和分页。

## 成功标准

- 两个或以上账号同房间对局时，当前行动玩家能看到倒计时，其他成员也能看到同一截止时间。
- 当前玩家超时后，服务端自动 `check` 或 `fold`，房间成员通过 socket 收到一致更新。
- 玩家断开 socket 后，其他成员看到离线状态；该玩家重连后恢复最新快照。
- 房间动作日志能展示从开局到结算的公开过程。
- 已结算手牌能从房间历史进入逐步回放。
- 房间战绩榜在新一手结算后更新。
- 房间成员可以发送文字和预设表情，非成员不能发送或订阅聊天。
