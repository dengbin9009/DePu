# 变更提案：HTTP 多人流程升级为 socket

## 背景

当前多人 v1 已经具备账号、钱包、建房、邀请码加入、入座、开局、轮流操作、手牌结果和个人战绩能力，但正式多人页面仍依赖 HTTP 周期轮询刷新房间状态，并通过 HTTP 接口提交开局和玩家动作。

现有前端在 `frontend/src/composables/useAppState.ts` 中使用 `startRoomPolling()` 每 2 秒调用房间详情、房间历史和当前手牌接口；正式手牌动作通过 `frontend/src/api/client.ts` 的 `startRoomHand()`、`fetchCurrentRoomHand()`、`submitRoomAction()` 调用 HTTP 路径完成。这个模型可以支撑本地验证，但多人对战会出现延迟感、重复请求、状态到达顺序不稳定和“别人操作后我不知道”的体验问题。

## 目标

- 将正式多人房间的实时状态同步升级为 socket 连接。
- 将正式多人开局和玩家动作提交升级为 socket command + ack。
- 后端在房间成员、座位、开局、动作推进、结算、钱包变化和错误发生时向相关连接广播事件。
- 前端进入房间后订阅对应 room channel，不再依赖 2 秒 HTTP 轮询刷新正式牌局。
- 保留 HTTP 作为账号、资料、钱包充值、建房、邀请码加入、历史查询和规则测试页能力的主要通道。

## 非目标

- 不实现真实支付。
- 不实现聊天、好友、大厅匹配、排行榜、机器人或托管。
- 不实现倒计时自动弃牌。
- 不移除独立规则测试页。
- 不把测试页 `/api/games/*` 调试设牌和回放能力迁移到正式多人 socket 协议。
- 不强制将所有 HTTP API 改造成 socket。

## 用户价值

- 玩家能近实时看到其他玩家入座、离座、开局和行动推进。
- 当前行动玩家提交动作后能得到明确 ACK，其他房间成员收到一致的状态广播。
- 多账号同房间操作时，前端不需要靠轮询猜测何时刷新。
- 断线重连后可以通过房间快照恢复，而不是等待下一次轮询。

## 风险

- socket 连接需要复用现有 Bearer token 鉴权，必须避免未授权用户订阅房间状态。
- 多连接并发提交动作时，必须保持规则引擎和存储事务的单一权威顺序。
- 前端状态更新从请求-响应变为事件驱动后，需要处理 ACK、广播、重连快照和错误事件的顺序。
- 结算后钱包余额、房间历史和个人战绩仍由持久化结果决定，不能只依赖内存广播。

## 成功标准

- 两个或以上真实账号进入同一房间后，入座、开局、行动推进和结算可以通过 socket 同步给房间成员。
- 当前行动玩家通过 socket 提交合法动作成功，非当前玩家通过 socket 提交动作收到 `not_your_turn`，且状态不改变。
- 前端正式牌桌不再依赖 `setInterval` 轮询当前手牌。
- 断开并重连后，客户端能够重新订阅房间并收到当前 room/hand 快照。
- 规则测试页和历史 HTTP 查询仍保持可用。
