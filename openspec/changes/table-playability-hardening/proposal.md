## Why

正式多人牌桌已经具备建房、入座、socket 对战、连续手牌、房主移交和牌谱回放等能力，但多轮真实验收仍反复出现控件遮挡、按钮无反馈、开局状态不收敛、牌面不可读和牌谱只有摘要等问题。当前需要把这些单点修复提升为稳定、可自动验证的产品契约，避免后续迭代再次破坏牌桌的基本可玩性。

## What Changes

- 建立牌桌响应式安全区和层级契约，保证顶部导航、座位、公共牌、底池、手牌、行动区、抽屉和底部工具栏在主要移动端、平板和桌面视口中互不遮挡。
- 加固关键交互入口，要求邀请好友、房主开局、牌桌开始、设置、聊天、战绩、牌谱、离桌和退出比赛均具备真实行为、明确禁用原因或可见错误反馈。
- 加固房间生命周期：手牌结算后按服务端权威状态自动进入下一手；人数不足时稳定回到等待态；房主退出后按座位顺序移交房主，无接管者时关闭房间。
- 统一清晰牌面组件，正面牌左上角显示点数，偏右下显示单个大花色，并在不同视口保持最低可读尺寸和红黑花色对比。
- 完善牌谱回顾，使已结算手牌自动加载并展示逐步动作、公共牌、底池和玩家公开状态，同时继续遵守隐藏底牌边界。
- 加固 socket 断线、重连和事件乱序处理，禁止旧手牌事件覆盖新手牌或已结算房间状态，未确认 command 不自动重放。
- 建立生产级质量门：后端 MySQL 测试、前端单元/类型/构建测试、多账号 socket 验收、响应式浏览器验收和回归报告全部通过后，任务才可标记完成。
- 不引入破坏性 API 变更；若新增 socket 版本或顺序字段，旧客户端缺失该字段时仍保持现有协议兼容。

## Capabilities

### New Capabilities

无。本次变更加固现有多人牌桌能力，不新增独立产品域。

### Modified Capabilities

- `multiplayer-poker`: 加强正式多人牌桌的响应式可玩性、关键交互反馈、连续手牌和房主生命周期、牌面可读性、牌谱明细、socket 恢复及生产验收要求。

## Impact

- 前端：`frontend/src/pages/RoomPage.vue`、`RoomInfoPage.vue`、牌桌抽屉组件、牌面展示组件或辅助函数、`useAppState.ts`、socket client、全局牌桌样式和前端测试。
- 后端：`backend/internal/api` 的房间/socket 事件处理、`backend/internal/storage` 的房主移交和结算状态写入、相关 MySQL 测试。
- 公共接口：沿用现有房间 HTTP API、`DELETE /api/rooms/{roomId}/members/me`、`room.leave`、`room.updated`、`hand.started`、`hand.updated`、`hand.settled` 和正式手牌 replay API；允许以向后兼容方式补充事件顺序元数据。
- 质量与文档：OpenSpec 任务、响应式浏览器测试脚本、生产验收报告和局域网开发启动说明。
