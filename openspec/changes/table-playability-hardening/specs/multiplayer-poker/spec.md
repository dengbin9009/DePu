## ADDED Requirements

### Requirement: 正式牌桌响应式安全区
正式多人牌桌 MUST 在代表性移动端、平板和桌面视口中为导航、座位、公共牌、底池、英雄手牌、行动区、抽屉和底部工具栏提供稳定安全区，任何装饰或次要信息 MUST NOT 阻断核心操作。

#### Scenario: 移动端进行中牌局无阻断遮挡
- **GIVEN** 360×640、390×844 或 430×932 的竖屏视口中存在进行中的正式手牌
- **WHEN** 当前玩家查看公共牌、自己的手牌和合法动作
- **THEN** 公共牌、底池、手牌、金额控件、行动按钮和底部工具栏均在可视区域内
- **AND** 这些核心区域互不发生阻断性覆盖
- **AND** 当前玩家可以点击每个可用行动按钮

#### Scenario: 九人桌座位信息保持边界
- **GIVEN** 房间配置为 9 人桌且昵称、在线状态和筹码同时展示
- **WHEN** 牌桌在代表性移动端或桌面视口中渲染
- **THEN** 每个座位节点保持稳定尺寸和位置
- **AND** 超长昵称使用截断或换行约束而不溢出座位
- **AND** 座位不得覆盖顶部导航、中心公共牌或英雄操作区

#### Scenario: 抽屉打开时牌桌继续接收实时更新
- **GIVEN** 用户打开聊天、战绩、牌谱或设置抽屉
- **WHEN** 房间收到新的 socket 房间或手牌事件
- **THEN** 底层权威状态继续更新
- **AND** 抽屉位于牌桌之上且其关闭按钮和主要控件可点击
- **AND** 关闭抽屉后用户看到最新牌桌状态

### Requirement: 关键牌桌操作必须有可观察结果
正式多人牌桌中的关键按钮 MUST 触发真实行为、显示明确禁用原因或呈现可见错误反馈；异步命令 MUST NOT 静默失败。

#### Scenario: 邀请好友复制或展示邀请码
- **GIVEN** 房间已经生成邀请码
- **WHEN** 用户点击“邀请好友”
- **THEN** 系统复制包含邀请码的邀请文本
- **OR** 在剪贴板不可用时直接展示可复制的邀请码
- **AND** 用户看到成功或降级反馈

#### Scenario: 房主从牌桌开始手牌
- **GIVEN** 当前用户是房主、房间处于等待态且满足开局条件
- **WHEN** 用户点击牌桌“开始”
- **THEN** 前端发送 `room.start_hand`
- **AND** 按钮在命令处理中避免重复提交
- **AND** 成功后牌桌展示服务端返回或广播的当前手牌

#### Scenario: 房主从房间信息页开始手牌
- **GIVEN** 当前用户在房间信息页且满足开局条件
- **WHEN** 用户点击“房主开局”
- **THEN** 系统使用与牌桌相同的开局 command 和错误处理
- **AND** 成功后返回对应牌桌并展示已开始手牌

#### Scenario: 开局失败提供可修正原因
- **GIVEN** 当前用户不是房主、人数不足、房间状态不允许或 socket command 被拒绝
- **WHEN** 用户尝试开局
- **THEN** 系统展示服务端返回的明确错误或禁用原因
- **AND** 不创建本地伪手牌
- **AND** 用户补足条件后可以再次尝试

#### Scenario: 未实现入口不伪装为可用
- **WHEN** 收藏、投诉、麦克风、保位离座或其他未实现入口出现在牌桌
- **THEN** 入口处于明确禁用状态或打开说明
- **AND** 点击不会静默无响应或写入虚假数据

### Requirement: 清晰且一致的正式牌面
正式多人牌桌 MUST 使用可读的统一牌面表示，正面牌左上角显示点数字母，偏右下显示单个大花色，并在代表性视口中维持最低可读尺寸和红黑对比。

#### Scenario: 英雄手牌可读
- **GIVEN** 当前用户收到两张或更多正式手牌
- **WHEN** 牌桌在最小支持视口中展示
- **THEN** 每张牌左上角显示 `A`、`K`、`Q`、`J`、`10` 或数字点数
- **AND** 偏右下只显示一个明显大于点数的花色
- **AND** 红桃和方块为高对比红色，黑桃和梅花为高对比深色
- **AND** 点数与花色不被牌框裁切

#### Scenario: 公共牌尺寸稳定
- **GIVEN** flop、turn 或 river 公共牌已经公开
- **WHEN** 公共牌数量从 3 张增长到 5 张
- **THEN** 公共牌保持固定宽高比和间距
- **AND** 不挤压底池、座位或房间水印到不可读状态

#### Scenario: 未知牌使用牌背
- **GIVEN** 手牌尚未发放或对应牌不应公开
- **WHEN** 牌桌需要展示占位
- **THEN** 系统展示稳定牌背
- **AND** 不展示可推断隐藏牌的文字或图案

### Requirement: 连续手牌与等待态收敛
正式多人房间 MUST 在一手牌结算后通过服务端权威状态收敛到下一手或等待态，MUST NOT 依赖某个前端保持在线才能继续。

#### Scenario: 人数足够时自动开始下一手
- **GIVEN** 一手正式手牌已经完成原子结算
- **AND** 房间仍开放且满足最少入座人数
- **WHEN** 结算展示窗口结束
- **THEN** 服务端在房间串行保护内重新校验最新状态
- **AND** 创建且只创建一手新的正式手牌
- **AND** 广播新 `hand.started`
- **AND** 所有订阅成员进入相同新手牌

#### Scenario: 人数不足时回到等待态
- **GIVEN** 一手牌结算后入座人数低于 `minPlayersToStart`
- **WHEN** 服务端尝试推进下一手
- **THEN** 房间保持 `waiting`
- **AND** `currentGameId` 为空
- **AND** 不广播虚假的 `hand.started`
- **AND** 房主看到可重新开始的等待态与人数不足提示

#### Scenario: 延迟自动开局不创建重复手牌
- **GIVEN** 结算后的自动开局任务尚未执行
- **AND** 期间房间被关闭、已有新手牌或条件发生变化
- **WHEN** 延迟任务获得房间锁
- **THEN** 服务端重新读取并校验权威状态
- **AND** 不覆盖已有手牌或创建第二手牌

#### Scenario: 结算事件顺序可恢复
- **WHEN** 一手牌结束
- **THEN** 服务端先完成手牌归档、钱包流水和房间状态持久化
- **AND** 再广播 `hand.settled`、钱包/战绩更新和 `room.updated`
- **AND** 重连用户能通过 `room.snapshot` 获得同一最终状态

### Requirement: 房主退出后的确定性移交
等待态房主退出房间时，系统 MUST 原子移交房主或关闭房间，并向剩余成员广播最终房间状态。

#### Scenario: 房主移交给下一位坐下玩家
- **GIVEN** 房主和至少一名其他玩家仍在等待态房间中坐下
- **WHEN** 房主通过 `room.leave` 或等效 HTTP 兜底退出房间
- **THEN** 系统释放房主座位并按房主座位之后的循环座位顺序选择下一位仍坐下玩家
- **AND** 原子更新 `rooms.owner_user_id` 与成员角色
- **AND** 广播包含新房主的 `room.updated`

#### Scenario: 普通玩家退出不改变房主
- **GIVEN** 退出者不是当前房主
- **WHEN** 该玩家退出等待态房间
- **THEN** 系统移除其成员和座位关系并按现有规则退还买入
- **AND** 当前房主保持不变

#### Scenario: 无人接管时关闭房间
- **GIVEN** 房主是房间内最后一位仍坐下玩家
- **WHEN** 房主退出等待态房间
- **THEN** 系统将房间置为 `closed`
- **AND** 清空当前手牌引用
- **AND** 后续开局 command 被明确拒绝

#### Scenario: 新房主权限实时生效
- **GIVEN** 房主移交事务已经成功
- **WHEN** 新房主客户端收到 `room.updated` 或重连快照
- **THEN** 新房主可以使用开始等房主专属操作
- **AND** 其他客户端不再把旧房主视为可执行房主操作的用户

### Requirement: 牌谱回顾展示具体历史步骤
牌谱回顾 MUST 直接展示已结算正式手牌的权威历史和逐步回放信息，不能只展示手牌数量或步骤数量。

#### Scenario: 打开牌谱自动加载最新一手
- **GIVEN** 当前房间存在至少一手已结算正式手牌
- **WHEN** 用户打开牌谱回顾抽屉
- **THEN** 系统展示按手牌编号排列的历史列表、赢家和底池摘要
- **AND** 自动请求并展示最新一手 replay steps

#### Scenario: 查看具体回放步骤
- **GIVEN** 正式回放接口返回多个 steps
- **WHEN** 用户选择某一步或点击上一动作、下一动作
- **THEN** 抽屉展示步骤序号、阶段、上一动作、底池、公共牌和玩家公开状态
- **AND** 切换到另一手时步骤索引重置为有效范围

#### Scenario: 回放保持隐藏牌边界
- **GIVEN** 某一步尚未公开某玩家底牌
- **WHEN** 用户查看该步骤
- **THEN** 抽屉仅展示服务端在该 step 返回的 `holeCards`
- **AND** 不使用最终归档数据补齐或推断隐藏底牌

#### Scenario: 回放加载失败可重试
- **WHEN** 正式回放请求失败、用户无权限或手牌不存在
- **THEN** 抽屉展示明确错误
- **AND** 保留历史列表和重试入口
- **AND** 不显示旧手牌的步骤冒充当前选择

### Requirement: socket 事件顺序与重连一致性加固
正式牌桌 MUST 拒绝会倒退本地状态的旧 socket 事件，并在重连后使用权威快照覆盖本地状态。未收到 ACK 的 command MUST NOT 自动重放。

#### Scenario: 核心事件携带向后兼容的顺序元数据
- **WHEN** 服务端发送 `room.snapshot`、`room.updated`、`hand.started`、`hand.updated` 或 `hand.settled`
- **THEN** 事件信封携带可选的正整数 `roomVersion`
- **AND** 手牌事件同时携带对应的 `handId` 和正整数 `handVersion`
- **AND** 同一手牌的新事件 `handVersion` 不小于旧事件
- **AND** 新字段不改变原有 payload 结构，忽略新字段的旧客户端仍可继续工作

#### Scenario: 旧手牌事件不能覆盖新手牌
- **GIVEN** 客户端已经接受新手牌的 `hand.started`
- **WHEN** 网络延迟导致上一手的 `hand.updated` 或 `hand.settled` 之后到达
- **THEN** 客户端忽略该旧手牌事件
- **AND** 当前手牌 id、行动位和合法动作保持新手牌状态

#### Scenario: 同一手牌旧步骤不能倒退状态
- **GIVEN** 客户端已经接受某手牌较新的动作序号或版本
- **WHEN** 更旧的 `hand.updated` 到达
- **THEN** 客户端不降低当前序号或恢复旧行动位

#### Scenario: 重连快照覆盖断线期间本地状态
- **GIVEN** 用户在等待态、进行中或结算窗口断开 socket
- **WHEN** 客户端重新连接并完成 `room.subscribe`
- **THEN** `room.snapshot` 覆盖对应房间的本地 room、hand、presence、日志、聊天和战绩摘要
- **AND** 页面展示与服务端当前状态一致

#### Scenario: 未确认 command 不自动重放
- **GIVEN** 客户端发送开局或行动 command 后断线且未收到 ACK
- **WHEN** 客户端重连并获得最新快照
- **THEN** 客户端不自动再次发送原 command
- **AND** 仅在最新快照仍允许时由用户主动重试

### Requirement: 生产级牌桌质量门
本变更的实现 MUST 通过自动化、协议、多账号和响应式浏览器质量门，发现的阻断问题 MUST 在任务完成前修复并重新验证。

#### Scenario: 自动化验证全部通过
- **WHEN** 实现准备标记完成
- **THEN** MySQL 环境下后端 `go test ./... -count=1` 通过
- **AND** 前端单元测试、类型检查和 production build 通过
- **AND** OpenSpec strict validate 与 `git diff --check` 通过

#### Scenario: 多账号完整生命周期验收
- **GIVEN** 至少两个真实账号进入同一房间
- **WHEN** 测试执行邀请加入、坐下、开局、完整一手、自动下一手、人数不足和房主退出移交
- **THEN** 每个客户端在无固定轮询条件下看到一致权威状态
- **AND** 钱包、历史、战绩和牌谱与数据库结果一致

#### Scenario: 响应式浏览器验收
- **WHEN** 测试在代表性移动端、平板和桌面视口运行等待态、行动态和抽屉场景
- **THEN** 自动检查核心元素矩形无阻断性重叠
- **AND** 检查可用按钮命中目标可点击
- **AND** 保存关键截图与结构化测试结果

#### Scenario: 失败阻止任务完成
- **WHEN** 任一质量门失败或浏览器发现新的阻断问题
- **THEN** 对应 OpenSpec task 保持未完成
- **AND** 修复后重新运行受影响测试与最终质量门
