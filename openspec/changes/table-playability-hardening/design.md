## Context

DePu 当前正式多人链路由 Go 后端规则引擎和 MySQL 提供权威状态，Vue 前端通过 socket command/事件展示房间和手牌。`migrate-http-to-socket`、`v1-1-table-experience` 与 `mockup-driven-table-experience` 已实现实时对战、倒计时、聊天、战绩、正式回放和移动牌桌，但真实浏览器验收暴露出一类共同问题：功能存在并不等于可玩。

问题跨越多个层次：牌桌页面的绝对定位区域互相侵占；按钮可能只有视觉外观而没有行为或错误反馈；结算、自动下一手和房主退出涉及数据库、socket 广播与前端事件顺序；牌谱接口返回完整步骤但抽屉只展示数量；牌面图片在移动视口中过小。测试也偏重源码 token 和单接口成功，缺少稳定的空间约束、连续多手、多账号乱序和端到端失败反馈验收。

约束如下：

- Go 规则引擎和 MySQL 仍是牌局、钱包、历史与房间生命周期的唯一权威来源。
- 正式牌桌不得恢复固定间隔 HTTP 轮询来掩盖 socket 状态问题。
- 正式回放不得泄露在对应步骤尚未公开的底牌。
- 独立规则测试页继续使用 `/api/games/*`，不得被本变更耦合。
- 本变更不得引入真实支付、好友关系、语音或新赛事系统。

## Goals / Non-Goals

**Goals:**

- 在 360×640、390×844、430×932、768×1024、1280×720 等代表性视口下保证牌桌核心区域可读、可点击且不发生阻断性遮挡。
- 使每个关键按钮都有真实 command/路由/弹层行为，或具备明确禁用状态和可见原因；异步失败不得静默。
- 将结算、等待、自动下一手、人数不足和房主退出定义为可重复验证的服务端生命周期。
- 通过事件顺序保护与重连快照，保证旧事件不能覆盖新手牌或最新房间状态。
- 统一牌面可读性，并让牌谱抽屉直接展示权威 replay steps。
- 建立适合 Loop Engineering 的小任务与分层退出门，每个任务可独立测试和审查。

**Non-Goals:**

- 不重写规则引擎、牌型比较、分池或钱包模型。
- 不新增真实好友邀请；“邀请好友”仅复制或展示房间邀请码/邀请文本。
- 不实现收藏、投诉、麦克风、保位离座、降落伞等完整后端流程。
- 不在已创建房间中动态切换长牌、短牌或奥马哈规则。
- 不把浏览器截图像素完全锁死为某一张效果图；验收关注安全区、可读性和操作可达性。

## Decisions

### 1. 用稳定安全区组织牌桌，不以单个截图的坐标作为布局真相

牌桌定义顶部导航区、桌面/座位区、中心公共牌区、英雄手牌区、行动区和底部工具区。共享 CSS 变量表达顶部与底部保留空间，固定格式元素使用明确尺寸、`aspect-ratio`、网格轨道和 `overflow` 约束。座位环只负责座位节点，中心水印必须 `pointer-events: none`，抽屉和模态框拥有独立最高交互层。

行动面板不得覆盖英雄手牌或底部工具栏；公共牌和底池不得与房间水印争夺同一视觉层；座位卡片必须截断长昵称且不能因在线状态或筹码变化改变尺寸。小屏幕无法同时展示次要元数据时，应降低次要信息密度，而不是缩小核心按钮和牌面到不可读。

备选方案是为每个视口写独立绝对坐标。该方案短期接近效果图，但容易在动态文案、9 人座和行动面板出现时再次重叠，因此不采用。

### 2. 所有关键操作统一为“可执行、可等待、可失败”三态

前端关键操作调用共享状态层方法，按钮在 command 未完成时进入 loading；成功后由 ACK/权威事件或一次明确的兜底读取收敛；失败时在当前操作上下文显示错误。房主开局页面成功后返回牌桌；邀请好友复制邀请码，剪贴板不可用时直接展示邀请码；尚未实现入口保持禁用并说明原因。

前端不得为了即时感自行构造 `hand.started`、修改 `currentSeat` 或结算结果。允许在 command ACK 后读取 `/current-hand` 作为显式一致性兜底，但不得恢复周期轮询。

备选方案是每个组件自行处理 socket 和错误。该方案会造成相同命令在牌桌页和房间信息页行为不一致，因此统一经 `useAppState`。

### 3. 房间生命周期由服务端状态机和房间级串行锁控制

状态转换采用以下语义：

1. `waiting` 且人数满足条件时，房主 `room.start_hand` 创建手牌并进入 `playing`。
2. 结算事务成功后先归档结果、更新钱包和房间，再广播 `hand.settled` 与 `room.updated`。
3. 默认在约 2 秒展示窗口后尝试自动下一手；房间级锁内重新读取最新房间、成员、座位和当前手牌。
4. 若仍为 `waiting`、无 `currentGameId` 且入座人数足够，则创建下一手并广播 `hand.started`。
5. 人数不足、房间关闭或已有新手牌时取消自动开局，保持权威状态不变。
6. 等待态成员退出使用 `room.leave` 或 HTTP 兜底。房主退出时按当前房主座位之后循环寻找仍坐下的下一位玩家；找到后原子更新 owner 和成员角色，找不到则将房间置为 `closed`。

自动下一手和房主移交不得由前端计时器决定。服务端重试前必须重新校验状态，避免延迟任务开启重复手牌。

### 4. socket 事件采用单调顺序保护，重连快照可覆盖本地状态

房间和手牌事件应携带或可推导单调顺序，例如房间 `version`、手牌 id + `seq`、服务端更新时间。客户端为当前房间保存最后接受的顺序：

- `room.snapshot`、`room.updated`、`hand.started`、`hand.updated` 和 `hand.settled` 的信封新增可选 `roomVersion`、`handId`、`handVersion`；旧客户端忽略新增字段即可继续工作。
- `roomVersion` 来自 MySQL `rooms.version`，房间成员、座位、房主、状态或当前手牌引用发生权威变更时原子递增。
- `handVersion` 复用已持久化的 `Game.Version`；`hand.started`、`hand.updated` 和 `hand.settled` 同时携带对应 `handId`，等待态快照不携带手牌版本。

- `room.snapshot` 在重连订阅后作为完整权威快照覆盖本地状态。
- 同一手牌中，仅接受不小于当前 `seq` 的事件。
- 新 `handId` 的 `hand.started` 可替换旧手牌；旧 `hand.updated`/`hand.settled` 不得覆盖新手牌。
- 房间已进入 `waiting` 或 `closed` 且 `currentGameId` 清空后，迟到的旧手牌事件不得恢复 `playing`。
- command 断线未收到 ACK 时不自动重放，用户根据新快照决定是否重试。

如果现有 payload 暂无显式版本，第一阶段可使用 `handId`、动作 `seq`、`updatedAt` 和客户端已知生命周期进行保护；最终应由服务端提供稳定版本字段。新增字段必须向后兼容。

### 5. 使用代码原生牌面组件保证最小可读性

正式多人牌桌的正面牌使用统一组件/渲染函数：左上角显示 `A/K/Q/J/10/9...`，偏右下只显示一个大号 `♠/♥/♦/♣`。红色花色使用高对比红色，黑色花色使用近黑色；牌框保持稳定宽高比。英雄手牌尺寸大于公共牌，代表性小屏下也必须达到设计规定的最小尺寸。

牌背使用稳定图案但不承载文字。历史页和规则测试页可逐步复用同一组件，但本变更首先保证正式牌桌和牌谱抽屉；不得改变牌码语义。

备选方案是继续缩放 PNG 完整牌面。小尺寸 PNG 同时缩小角标和中心图案，无法满足可读性要求，因此正式牌桌不再依赖它作为主要牌面。

### 6. 牌谱抽屉直接消费正式回放接口，不复制规则

打开有历史的牌谱抽屉时自动加载最新一手 `GET /api/rooms/{roomId}/hands/{handId}/replay`。抽屉展示手牌列表、赢家/底池摘要、当前步骤、阶段、动作、公共牌、每名玩家公开状态和上一/下一动作。切换手牌重置步骤索引；请求失败显示可重试错误。

回放数据完全来自服务端 `steps`。前端只做状态和动作标签格式化，不重新计算底池、赢家或牌型。未到公开步骤的底牌继续为空；非成员权限保持后端拒绝。

### 7. 质量门分为自动化、协议、多账号和视觉四层

每个 Loop task 先增加能失败的测试，再实现并完成局部验证。整体完成必须依次通过：

- 后端：`go test ./... -count=1`，使用 MySQL 临时 database。
- 前端：Vitest、`vue-tsc --noEmit`、Vite production build、`git diff --check`。
- 协议：鉴权、非成员、非房主、非当前玩家、重复 command、结算事件顺序、自动下一手、人数不足、房主移交、重连和旧事件边界。
- 浏览器：至少两个真实账号，覆盖等待、入座、开局、完整一手、自动下一手、退出移交、牌谱步骤、断线恢复，以及代表性视口的重叠和可点击性。

浏览器测试必须检查元素矩形与点击命中，不只依赖截图肉眼判断；同时保留关键截图和结构化结果。测试中发现的问题在同一任务内修复并重新验收。

## Risks / Trade-offs

- [绝对定位历史样式较多，安全区改造可能影响多视口] → 先建立布局度量测试，再按区域渐进调整；禁止一次性重写整个页面。
- [自动下一手延迟任务可能与玩家退出、房间关闭或手动开局竞争] → 所有转换进入房间级串行锁，并在执行时重新读取权威状态。
- [新增事件版本字段可能影响旧客户端] → 只新增可选字段，服务端仍发送现有字段；客户端对缺失版本提供保守兼容路径。
- [源码 token 测试无法证明真实可点击] → 保留组件测试，但把浏览器命中测试和多账号 E2E 作为完成门。
- [小屏同时容纳 9 人、手牌和操作区会牺牲次要信息] → 优先保证牌、轮到谁、筹码和行动按钮；网络延时、房间长名称等次要信息允许截断或收纳。
- [回放展示全部底牌可能违反隐私] → 只渲染服务端当前 step 明确返回的 `holeCards`，不从归档参与者数据补齐隐藏牌。

## Migration Plan

1. 先补充事件顺序、生命周期和布局度量测试，建立当前行为基线。
2. 后端按向后兼容方式加固房间状态转换、房主移交和事件版本，不做破坏性 schema 删除。
3. 前端增加旧事件过滤、统一操作反馈和牌面组件，再逐区调整牌桌安全区。
4. 完善牌谱抽屉与响应式 E2E，使用两账号跑连续多手和退出移交。
5. 全量验证通过后提交；部署时先后端后前端，旧前端仍可消费新增字段前的既有事件结构。
6. 若出现回归，前端可回滚到旧布局，后端新增字段与测试保持兼容；自动下一手可通过配置关闭并退回房主手动开始，不回滚已持久化结果。

## Open Questions

- 自动下一手展示窗口是否固定为 2 秒，还是后续做成房间配置；本变更默认 2 秒并允许服务端常量配置。
- 房主移交在“下一位坐下玩家”之外是否需要考虑未坐下成员；本变更只选择仍坐下玩家，无人接管则关闭房间。
- 正式牌面组件是否同步替换历史页和规则测试页；本变更必须覆盖正式牌桌与牌谱抽屉，其他页面可在不扩大风险时复用。
