# multiplayer-poker 增量规格：mockup-driven-table-experience

## ADDED Requirements

### Requirement: 效果图式创建比赛配置

系统 MUST 提供面向房主的创建比赛体验，使房主可以按移动端效果图配置训练赛牌桌，并在创建成功后进入对应房间。

#### Scenario: 创建短牌训练赛

- **GIVEN** 用户已登录且拥有创建房间所需金币
- **WHEN** 用户选择训练赛、短牌玩法、输入比赛名称并选择 Ante、最小带入、带入上限、训练时长和单桌最大人数后确认创建
- **THEN** 系统创建正式多人房间
- **AND** 房间详情包含比赛名称、比赛模式、玩法、Ante、买入范围、训练时长、座位数和邀请码
- **AND** 用户进入该房间牌桌

#### Scenario: 不可用玩法或模式不会误创建

- **GIVEN** 某比赛模式或玩法尚未完整支持
- **WHEN** 用户尝试选择或提交该配置
- **THEN** 系统阻止创建或返回明确错误
- **AND** 不生成不可用房间

#### Scenario: 创建参数越界

- **WHEN** 用户提交低于最小值、高于最大值或彼此冲突的配置
- **THEN** 系统返回明确字段错误
- **AND** 不创建房间

### Requirement: 商城金币页与模拟充值入口

系统 MUST 提供商城页面作为金币补充入口，并复用现有模拟充值语义。未实现的资产和权益分类 MUST 明确展示为占位或不可用状态。

#### Scenario: 查看金币商城

- **GIVEN** 用户已登录
- **WHEN** 用户进入商城并停留在金币分类
- **THEN** 系统展示当前金币余额、钻石/积分/普通卡概览和金币充值档位
- **AND** 钻石、VIP 卡、装扮、道具、课程分类不会触发未实现交易

#### Scenario: 模拟充值成功

- **GIVEN** 用户在商城选择有效金币档位
- **WHEN** 用户确认模拟充值
- **THEN** 系统增加金币余额
- **AND** 写入 `recharge_simulated` 钱包流水
- **AND** 商城资产概览刷新为最新余额

#### Scenario: 从买入弹窗跳转商城后返回

- **GIVEN** 用户在牌桌补充记分牌时金币不足
- **WHEN** 用户从弹窗进入商城并完成模拟充值
- **THEN** 用户可以返回原牌桌上下文
- **AND** 再次打开或继续买入时显示最新可用金币

### Requirement: 牌桌内补充记分牌买入

系统 MUST 在牌桌内提供补充记分牌弹窗，让用户坐下前选择合法带入金额，并根据房间规则和钱包余额完成入座。

#### Scenario: 点击空座打开买入弹窗

- **GIVEN** 用户已加入房间但尚未入座
- **WHEN** 用户点击空座或坐下入口
- **THEN** 系统打开补充记分牌弹窗
- **AND** 弹窗展示可选带入档位、当前带入值、消耗金币、可用金币、取消和确定操作

#### Scenario: 确认合法买入

- **GIVEN** 用户选择的带入金额满足房间最小带入、带入上限和钱包余额要求
- **WHEN** 用户确认买入
- **THEN** 系统占用对应座位
- **AND** 扣减或冻结对应金币语义按现有钱包规则执行
- **AND** 牌桌座位状态更新为该用户已坐下

#### Scenario: 非法买入被拒绝

- **WHEN** 用户选择低于最小带入、高于带入上限、超过钱包余额的金额，或目标座位已被占用
- **THEN** 系统拒绝买入
- **AND** 弹窗展示明确错误
- **AND** 房间座位状态不改变

### Requirement: 移动端牌桌主舞台

正式多人牌桌 MUST 提供移动端竖屏优先的游戏桌布局，并继续以服务端房间、手牌和 socket 事件为权威来源。

#### Scenario: 等待开局牌桌

- **GIVEN** 房间已创建且尚未开始当前手牌
- **WHEN** 房间成员进入牌桌
- **THEN** 牌桌展示顶部健康提示、网络延时、座位环、房间模式、玩法、手数、Ante、级别、房号和邀请码
- **AND** 空座展示可坐下入口
- **AND** 底部工具栏展示设置、离桌或返回、聊天和麦克风占位

#### Scenario: 房主等待态操作

- **GIVEN** 当前用户是房主且房间满足开局条件
- **WHEN** 用户查看等待态牌桌
- **THEN** 用户可以看到并使用开始按钮
- **AND** 可以看到解散比赛和邀请好友入口

#### Scenario: 非房主等待态限制

- **GIVEN** 当前用户不是房主
- **WHEN** 用户查看等待态牌桌
- **THEN** 系统不允许该用户发起房主专属开局或解散操作
- **AND** 若用户尝试绕过界面提交房主操作，系统返回明确错误且不改变房间状态

#### Scenario: 已坐下玩家信息

- **GIVEN** 用户已经在牌桌入座
- **WHEN** 用户查看牌桌底部玩家区域
- **THEN** 系统展示用户头像或占位头像、昵称、等待/行动状态和当前筹码
- **AND** socket 更新后该区域随权威状态刷新

### Requirement: 牌桌内聊天记录面板

正式多人牌桌 MUST 将房间聊天展示为牌桌内底部抽屉，并复用现有房间聊天权限、长度和频率限制。

#### Scenario: 打开聊天记录

- **GIVEN** 用户在正式多人牌桌内
- **WHEN** 用户点击聊天入口
- **THEN** 系统打开聊天记录底部抽屉
- **AND** 抽屉展示常用语入口、聊天记录标题、关闭按钮、最近消息列表、输入框和发送按钮

#### Scenario: 发送合法聊天

- **GIVEN** 用户是房间成员
- **WHEN** 用户输入不超过限制的文本并发送
- **THEN** 服务端接受消息并向房间成员广播
- **AND** 聊天抽屉展示该消息

#### Scenario: 非法聊天不进入记录

- **WHEN** 用户发送空文本、过长文本、频率过快消息，或非成员尝试发送
- **THEN** 系统返回明确错误
- **AND** 不把该消息写入房间聊天记录

### Requirement: 当前战绩牌桌抽屉

正式多人牌桌 MUST 提供当前战绩抽屉，展示本房间内玩家的带入、手数和战绩，并在无数据时提供明确空状态。

#### Scenario: 查看当前战绩

- **GIVEN** 用户在正式多人牌桌内
- **WHEN** 用户打开当前战绩抽屉
- **THEN** 系统展示标题“当前战绩”、剩余时间、昵称、带入、手数和战绩列
- **AND** 数据以房间战绩榜和房间座位/买入信息为来源

#### Scenario: 当前战绩空状态

- **GIVEN** 房间内尚无可展示战绩
- **WHEN** 用户打开当前战绩抽屉
- **THEN** 系统展示观众人数或暂无战绩空状态

#### Scenario: 结算后战绩刷新

- **GIVEN** 一手正式手牌结算成功
- **WHEN** 服务端广播房间战绩榜更新或用户刷新战绩
- **THEN** 当前战绩抽屉展示最新手数和输赢

### Requirement: 牌谱回顾面板

正式多人牌桌 MUST 提供牌谱回顾面板，用于查看已结算手牌的公开回放摘要，并保持正式多人回放与规则测试页回放隔离。

#### Scenario: 无牌谱数据

- **GIVEN** 当前房间没有可回顾的正式手牌
- **WHEN** 用户打开牌谱回顾面板
- **THEN** 系统展示“暂无任何数据”
- **AND** 底部控制区保持可见但不允许执行无效回放

#### Scenario: 查看正式手牌回放

- **GIVEN** 当前房间存在已结算正式手牌
- **WHEN** 用户打开牌谱回顾并选择回放
- **THEN** 系统展示该手牌的公开 replay steps
- **AND** 支持上一手、下一手、播放或逐步查看
- **AND** 不泄露未公开底牌

#### Scenario: 占位入口不执行未实现流程

- **WHEN** 用户点击收藏或投诉等本次未完整实现入口
- **THEN** 系统展示占位或禁用提示
- **AND** 不写入虚假的收藏或投诉记录

### Requirement: 牌桌设置菜单

正式多人牌桌 MUST 提供底部设置菜单，集中承载桌面设置、站起围观、带入记分牌、比赛设置、短牌规则、退出比赛和若干占位说明入口。

#### Scenario: 打开牌桌设置

- **GIVEN** 用户在牌桌内
- **WHEN** 用户点击底部设置入口
- **THEN** 系统打开牌桌设置菜单
- **AND** 菜单展示桌面设置、站起围观、带入记分牌、比赛设置、短牌规则、保位离座、降落伞说明和退出比赛

#### Scenario: 站起围观

- **GIVEN** 用户已入座且当前房间允许离座
- **WHEN** 用户选择站起围观
- **THEN** 系统释放该用户座位
- **AND** 用户仍可留在房间内旁观

#### Scenario: 退出比赛

- **GIVEN** 用户在房间内
- **WHEN** 用户选择退出比赛并确认
- **THEN** 系统按房间离开规则处理成员和座位
- **AND** 用户返回大厅或上一层入口

#### Scenario: 未实现菜单项

- **WHEN** 用户点击保位离座、降落伞说明或桌面设置中尚未完整实现的能力
- **THEN** 系统展示说明、禁用状态或本地占位
- **AND** 不改变正式手牌状态

### Requirement: 现有多人能力与规则测试页保持兼容

本次体验优化 MUST NOT 破坏现有 socket 多人对战、钱包、聊天、战绩榜、正式回放和独立规则测试页。

#### Scenario: socket 对战继续可用

- **GIVEN** 两个或以上账号进入同一房间
- **WHEN** 玩家入座、开局、行动、结算
- **THEN** 房间成员继续通过 socket 收到一致状态
- **AND** 钱包、历史和战绩保持一致

#### Scenario: 规则测试页继续隔离

- **WHEN** 测试者进入独立规则测试页创建测试牌局、调试设牌或执行只读回放
- **THEN** 测试页继续按 `/api/games/*` 语义工作
- **AND** 正式多人牌桌不会收到这些测试页事件
