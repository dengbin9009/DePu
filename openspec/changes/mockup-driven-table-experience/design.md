# 技术设计：mockup-driven-table-experience

## 总体策略

本次变更是“体验整合 + 少量资源模型扩展”，不是重新实现牌局规则。后端规则引擎、房间状态、socket 事件、钱包和历史结果继续作为权威来源；前端负责把这些能力组织成效果图所示的移动端游戏桌体验。

设计分为四层：

- 创建与配置：从大厅进入“创建比赛”，用档位控件生成房间配置。
- 资产与买入：商城承载模拟充值，牌桌内买入弹窗承载带入记分牌。
- 牌桌主舞台：移动端座位环、房间信息、房主按钮、底部工具栏。
- 牌桌面板：聊天记录、当前战绩、牌谱回顾、牌桌设置等覆盖面板。

## 房间配置模型

现有房间创建至少支持 `ruleSetId`、`seatCount` 和 `minPlayersToStart`。效果图中的参数需要分级处理：

- 首批必须落地并写入房间或规则配置：玩法类型、房间名称、Ante 档位、最小带入、带入上限、训练时长、单桌最大人数。
- 可先映射到现有语义：短牌使用已有 ruleset；单桌最大人数映射 `seatCount`；最小开局人数沿用默认 2；最小带入和带入上限用于入座校验。
- 暂不完整落地：SNG、级别升级、奥马哈。UI 可展示受限入口，但不得承诺完整赛事能力。

建议房间响应增加只读展示字段，便于牌桌中心区域稳定渲染：

- `name`
- `mode`: `training` 或未来 `sng`
- `variant`: `short_holdem`、`holdem`、未来 `omaha`
- `ante`
- `minBuyIn`
- `maxBuyIn`
- `buyInCap`
- `durationMinutes`
- `level`
- `handNo`

如果后端暂不全部持久化，前端不得伪造影响服务端校验的字段；可先由服务端返回默认配置。

## 创建比赛页

创建比赛页使用移动端单列布局：

- 顶部返回与标题“创建比赛”。
- 比赛类型切换：训练赛默认可用，SNG 显示占位或禁用说明。
- 玩法切换：短牌默认可用；国际扑克按现有 holdem 能力可用；奥马哈先禁用。
- 房间名称输入。
- 多个档位选择器：Ante、最小带入、带入上限、训练时长、单桌最大人数。
- 固定底部“确定创建”按钮。

所有提交前校验应以服务端为准。前端可以做范围提示，但不能绕过服务端错误。

## 商城与钱包

商城页面作为现有模拟充值的产品化入口：

- 顶部返回与标题“商城”。
- 分类标签：金币可用；钻石、VIP 卡、装扮、道具、课程先作为占位。
- 资产概览：金币余额来自钱包；钻石、积分、普通卡状态可以显示默认值或未激活状态。
- 金币商品卡：使用服务端充值档位；若服务端未返回价格和赠送信息，可前端按模拟档位展示“模拟充值”文案，不得误导为真实支付。

充值确认后调用现有模拟充值能力，成功后刷新钱包，并支持返回此前的牌桌或买入弹窗。

## 补充记分牌弹窗

坐下流程改为牌桌内弹窗：

1. 用户点击空座或底部坐下入口。
2. 系统打开“补充记分牌”弹窗，展示房间允许的带入档位。
3. 用户选择带入记分牌数量。
4. 弹窗展示消耗金币、可用金币和金币不足状态。
5. 用户确认后提交占座买入。
6. 成功后关闭弹窗，并等待 socket 或 HTTP 返回后的房间状态更新。

金币不足时，弹窗提供“前往商城购买金币”入口；从商城返回后应保留原房间上下文。

## 牌桌主舞台

牌桌页应优先服务移动端竖屏，保持以下稳定区域：

- 顶部安全提示和网络延时。
- 座位环：最多 9 人展示；不同 seatCount 时仍保持可读位置。
- 中心房间信息：玩法、手数、房间名或品牌、Ante、级别、房号、邀请码。
- 等待态操作：房主可看到解散比赛、邀请好友、开始；非房主看到等待或邀请信息。
- 玩家底部区域：未坐下时显示坐下入口；已坐下时显示头像、昵称、等待/行动状态和筹码。
- 底部工具栏：设置/菜单、离桌或返回、聊天、麦克风占位。

牌桌不得用前端本地状态乐观推进手牌；手牌、座位、战绩、聊天和回放仍以服务端快照和事件为准。

## 牌桌面板

### 当前战绩

当前战绩作为左侧抽屉或覆盖面板展示：

- 标题“当前战绩”。
- 训练赛剩余时间。
- 表头：昵称、带入、手数、战绩。
- 房间玩家战绩来自现有房间战绩榜，带入可来自座位买入或参与者记录。
- 无玩家或仅旁观时显示观众人数空状态。

### 聊天记录

聊天记录作为底部抽屉展示：

- 左侧“常用语”标签。
- 中间标题“聊天记录”。
- 右侧关闭按钮。
- 消息列表展示最近聊天与表情。
- 底部输入框限制 40 个汉字或等效长度，发送按钮调用现有 `chat.send`。

### 牌谱回顾

牌谱回顾作为右侧或中央覆盖面板展示：

- 标题“牌谱回顾 - 房号/手牌号”。
- 回放按钮。
- 空状态“暂无任何数据”。
- 底部进度条、收藏占位、上一手、下一手、当前第几手、投诉占位。
- 有回放数据时复用正式多人 replay steps；无数据时显示空状态。

收藏和投诉本次仅作为占位或禁用入口，不新增完整后端流程。

### 牌桌设置

牌桌设置作为底部菜单展示：

- 桌面设置：可先作为占位或本地显示偏好入口。
- 站起围观：对应离座但仍留在房间。
- 带入记分牌：打开补充记分牌弹窗；若已在牌局中，按服务端允许范围处理。
- 比赛设置：展示当前房间配置。
- 短牌规则：展示静态规则说明或跳转规则说明面板。
- 保位离座、降落伞说明：占位说明。
- 退出比赛：离开房间或返回大厅，按现有房间离开规则执行。

## 错误与边界

- 房主专属动作对非房主必须禁用或返回明确错误。
- 房间已开始时，不能允许用户修改会影响当前手牌公平性的配置。
- 买入金额低于最小带入、高于上限、余额不足或座位被占时，必须显示明确错误并保持弹窗可修正。
- 商城充值失败时保留用户上下文，不清空牌桌状态。
- 面板打开期间收到 socket 更新时，底层状态必须继续更新；关闭面板后看到最新牌桌。

## 测试策略

后端至少覆盖：

- 创建房间配置字段的默认值、范围校验和响应展示。
- 买入金额低于最小带入、高于上限、余额不足、座位冲突。
- 房间配置不影响现有开局、行动、结算和历史结果原子性。
- 商城模拟充值仍写入钱包流水。

前端至少覆盖：

- 创建比赛页按档位提交正确配置。
- 补充记分牌弹窗的档位选择、金币不足、前往商城和确认买入。
- 牌桌等待态、已坐下态、房主/非房主按钮差异。
- 聊天、当前战绩、牌谱回顾、牌桌设置面板打开关闭与状态更新。
- 商城金币页充值后刷新余额并能返回原上下文。
- 移动端主要视口无文字重叠、底部工具栏不遮挡核心操作。

手动验收至少覆盖：

- 创建短牌训练赛，邀请另一个账号加入并坐下。
- 金币不足时从坐下弹窗跳商城模拟充值，再返回完成坐下。
- 房主开始手牌，两端继续通过 socket 同步行动和结算。
- 打开聊天、当前战绩、牌谱回顾、牌桌设置，关闭后仍停留同一牌桌。
- 独立规则测试页仍可进入并完成原有测试流程。
