# 研究记录：德州扑克完整牌局模拟器

## 决策：前后端分离但规则集中在 Go 后端

**选择**: Vue 前端只负责可视化、输入和展示，Go 后端作为规则权威，提供 HTTP JSON API。

**理由**: 德州扑克规则包含最小加注、短筹码强制投入、不足额全下、边池、短牌牌型排序、只读回放和调试牌锁定等高风险逻辑。集中在后端可以保证前端、回放和测试使用同一套规则。

**备选方案**:

- 前端也实现规则推断：被拒绝，因为会导致双端规则漂移。
- 纯前端本地模拟：被拒绝，因为 v1 要 SQLite 保存历史和回放。

## 决策：用 `RuleSet` + `BettingStructure` 表达长牌和短牌差异

**选择**: `RuleSet` 负责牌组、顺子、牌型排序和可选下注结构；`BettingStructure` 负责 `blinds` 或 `ante` 的金额、强制投入和行动顺序。

**理由**: 长牌和短牌差异不仅是牌组数量，还包括短牌 A-6-7-8-9 顺子、短牌默认同花大于葫芦，以及短牌可选择 `blinds` 或 `ante + buttonBlind`。拆分规则集和下注结构可以避免到处写条件分支。

**备选方案**:

- 用布尔字段 `isShortDeck`：被拒绝，因为后续新增规则变体会扩散条件判断。
- 为长牌和短牌写完全独立状态机：被拒绝，因为投注、底池、回放大部分逻辑可复用。

## 决策：创建牌局校验由后端统一执行

**选择**: 后端校验 2 到 10 人、`seatNo` 为 `1..N` 连续整数、`buttonSeat` 对应已有座位、玩家名称非空且本局唯一、筹码为正整数、`smallBlind < bigBlind`、`ante` 和 `buttonBlind` 为正整数。

**理由**: 这些规则影响按钮位、行动顺序、强制投入和行动历史。由后端统一校验可以让 API 测试和前端表现一致，并符合宪章中“不能静默修正用户输入”的要求。

**备选方案**:

- 前端负责座位号和玩家名校验：被拒绝，因为 API 仍可能收到非法请求。
- 允许跳号座位：被拒绝，因为 v1 没有空座管理需求，连续座位能简化行动顺序和牌桌显示。
- 固定盲注金额：被拒绝，因为用户需要可配置下注结构。

## 决策：短筹码强制投入按实际剩余筹码全下

**选择**: 玩家筹码不足以完整支付小盲、大盲、`ante` 或 `buttonBlind` 时，系统扣除该玩家全部剩余筹码，设置为全下，并按实际投入额参与主池和边池。

**理由**: 这更符合德州扑克规则，也能复用 v1 已要求支持的全下、主池/边池和结算逻辑。拒绝创建会让模拟器无法覆盖常见边界场景。

**备选方案**:

- 创建时拒绝短筹码玩家：被拒绝，因为缺失规则边界。
- 自动补足筹码：被拒绝，因为会静默改变用户输入。

## 决策：调试指定牌允许部分指定，后端随机补足剩余牌

**选择**: 调试模式下，在首个玩家动作前可以指定全部或部分玩家手牌与公共牌；已指定牌固定，未指定手牌和后续公共牌由后端从排除已指定牌后的合法牌堆随机补足。

**理由**: 规则调试经常只需要固定关键牌或部分公共牌。部分指定能快速复现特殊牌型、边池和平分场景，同时保持随机模式的便利性。

**备选方案**:

- 要求指定所有玩家手牌和 5 张公共牌：被拒绝，因为调试成本过高。
- 未指定牌保持未知，推进时再要求补填：被拒绝，因为会打断牌局推进流程。
- 首个动作后允许重置：被拒绝，因为会破坏行动历史和回放可解释性。

## 决策：SQLite 保存权威状态和行动日志，回放只读

**选择**: SQLite 保存牌局快照、行动日志、下注结构、调试牌信息和摊牌结果。回放接口只返回指定动作后的只读快照，不改变当前权威状态，也不能作为后续动作的版本依据。

**理由**: 本地单机产品不需要服务器级数据库，SQLite 足够轻量。只读回放避免历史节点继续行动造成分支状态，也让行动历史保持线性可解释。

**备选方案**:

- 只保留内存状态：被拒绝，因为刷新后恢复和回放是 v1 需求。
- 回放节点可继续行动：被拒绝，因为会引入分支历史和冲突合并。

## 决策：回放动作序号采用显式边界

**选择**: `toSeq=0` 返回初始发牌后的只读快照；`toSeq` 小于 0 或大于最新动作序号时返回错误，不自动修正。

**理由**: `toSeq=0` 便于复盘初始局面；越界返回错误符合宪章中“不静默修正用户输入”的要求，也让前端和测试能明确识别请求问题。

**备选方案**:

- 越界时自动调整到最近有效节点：被拒绝，因为会隐藏调用方错误。
- 不支持初始状态回放：被拒绝，因为初始强制投入和发牌状态是复盘关键节点。

## 决策：牌桌配件和扑克动画只作为前端表现层

**选择**: Vue 前端负责将后端快照渲染成接近真实牌桌的视觉体验，包括桌面、座位、手牌、公共牌、筹码/底池、按钮位、当前行动者和动作控制；发牌、公共牌出现、行动切换和底池变化使用短暂动画表达。牌桌视觉覆盖 2 到 10 人，4 人桌在 1280px 桌面第一屏完整可见，6 到 10 人允许更紧凑布局或移动端纵向滚动，但不得关键重叠。

**理由**: 用户明确要求当前文字型界面升级为可视化辅助。牌桌和动画能显著提升演示、复盘和规则理解效率，但规则权威仍必须保持在 Go 后端，避免前端动画状态影响下注、发牌或结算。

**备选方案**:

- 保持文字列表界面：被拒绝，因为不满足“合理的牌桌配件和扑克动画”的可视化目标。
- 前端根据动画自行推进阶段：被拒绝，因为会让展示层变成规则来源，破坏后端权威。
- 使用长时间或阻塞式动画：被拒绝，因为动作提交和状态阅读必须保持流畅。
- 只验收 4 人牌桌：被拒绝，因为规则层支持 2 到 10 人，后续人数扩展会造成布局返工。

## 决策：回放只做手动节点切换，不做自动播放时间轴

**选择**: v1 回放继续以只读快照为核心，操作者手动选择动作节点查看对应牌桌状态；前端可以在节点切换时展示短暂过渡动画，但不实现整手牌自动播放时间轴。

**理由**: 这保留了“可复盘”的核心价值，同时避免引入独立播放控制、时间轴速度、暂停/恢复和动画排队状态机。回放仍然只依赖后端快照和行动历史，不改变权威状态。

**备选方案**:

- 自动播放整手牌：被拒绝，因为会显著增加 v1 前端状态机和验收复杂度。
- 完全静态切换：被拒绝，因为用户已经要求扑克动画，可视化回放节点切换需要最小过渡反馈。

## 决策：v1 保持单机本地边界

**选择**: v1 不实现登录、真钱结算、多人在线同步、房间匹配、排行榜、AI 行动建议或 GTO 求解。

**理由**: 当前核心价值是规则模拟、调试、保存和回放。引入账号、资金或实时同步会显著增加安全、合规和并发复杂度。

**备选方案**:

- 预留轻量账号系统：被拒绝，因为会偏离 v1 单机边界。
- 加入 AI 建议：被拒绝，因为会混淆规则引擎和策略引擎。
