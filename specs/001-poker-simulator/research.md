# 研究记录：德州扑克完整牌局模拟器

## 决策 1：使用平衡分层架构

**决策**: 采用 Vue 前端、Go API、Go 规则引擎、SQLite 持久化的平衡分层架构。

**原因**: 完整牌局模拟器需要同时具备可操作界面和严谨规则判定。前端只负责展示与交互，后端作为规则权威，可以避免状态分叉。

**替代方案**:

- 引擎优先：规则更纯粹，但第一版可见产品体验较慢。
- 界面优先：演示更快，但容易把规则散落在前端，后续修正成本高。

## 决策 2：v1 同时支持长牌和短牌

**决策**: 通过 `RuleSet` 抽象支持长牌 52 张和短牌 36 张。短牌默认支持 A-6-7-8-9，且同花大于葫芦。

**原因**: 用户明确要求第一版同时支持长牌和短牌。规则集抽象可以控制复杂度，避免未来扩展时重写核心逻辑。

**替代方案**:

- 先做长牌：实现更快，但无法满足用户当前目标。
- 先做短牌：会降低标准德州扑克验证价值。

## 决策 3：支持随机发牌和调试指定牌

**决策**: 默认随机洗牌发牌，同时提供调试模式手动指定手牌和公共牌。

**原因**: 真实流程需要随机发牌；规则验证需要快速构造特殊场景，例如短牌顺子、同花大于葫芦、多人全下和边池。

**替代方案**:

- 仅随机：更贴近真实牌局，但难以测试边界。
- 仅手动：适合规则调试，但不像完整模拟器。

## 决策 4：使用 SQLite 保存牌局

**决策**: v1 使用 SQLite 保存牌局快照、行动历史和回放数据。

**原因**: SQLite 足够支持本地单机工具，部署简单，且能验证保存和回放能力。

**替代方案**:

- 不保存：实现简单，但刷新后丢失，复盘价值低。
- 浏览器本地保存：后端无法作为完整权威来源，调试 API 和回放能力受限。

## 决策 5：不做账户、真钱和多人在线

**决策**: v1 明确排除登录、真钱、多房间、多人实时同步、排行榜和 AI 建议。

**原因**: 这些能力会引入身份、权限、资金安全、同步一致性和合规问题，和第一版规则验证目标不一致。

**替代方案**:

- 预留账户接口：会污染 v1 数据模型和任务拆分。
- 简单多人同步：会显著增加状态冲突和网络同步复杂度。
