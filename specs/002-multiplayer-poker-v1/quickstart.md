# 快速开始：多人德州扑克对战 v1

> 本文档用于定义未来实现完成后的手动验收流程。本轮仅编写文档，不要求当前仓库已经具备全部能力。

## 目标

验证以下多人产品主路径与测试页边界：

- 账号注册、登录与昵称唯一性
- 模拟充值与钱包流水
- 建房、邀请码加入、入座与开局
- 多真实账号轮流操作一手正式牌局
- 每手牌结果、个人战绩和钱包一致性
- 独立规则测试/回放页面仍可正常使用

## 前置条件

- 已准备好后端服务和前端页面的未来实现版本
- 生产模式可连接 MySQL
- 开发/测试模式可切换到 SQLite
- 系统内不存在以下预置用户名或昵称：`owner01`、`player02`、`player03`

## 场景 1：账号与昵称

1. 注册账号 `owner01`，设置密码。
2. 注册账号 `player02`，设置密码。
3. 注册账号 `player03`，设置密码。
4. 依次登录三个账号，分别设置昵称 `房主A`、`玩家B`、`玩家C`。
5. 尝试把 `player03` 的昵称改为 `房主A`。
6. 验证系统拒绝并返回昵称重复错误。

**预期结果**:

- 三个账号均可成功注册和登录。
- 三个昵称可被正确展示。
- 重复昵称修改请求失败，原昵称保持不变。

## 场景 2：模拟充值与钱包流水

1. 使用 `owner01` 登录。
2. 打开充值页面，查看固定金币档位。
3. 选择一个档位并确认充值。
4. 再选择第二个档位并确认充值。
5. 查看钱包余额和流水列表。

**预期结果**:

- 充值前先有明确确认提示。
- 每次确认后都直接模拟成功。
- 余额按两次档位金额累加。
- 流水中有两条 `recharge_simulated` 记录，且金额与余额变更一致。

## 场景 3：建房、加入与入座

1. 使用 `owner01` 创建房间。
2. 记录系统返回的邀请码。
3. `player02` 与 `player03` 分别使用邀请码加入房间。
4. 三名玩家分别选择不同座位入座。
5. 让 `player03` 尝试占用已被 `player02` 使用的座位。

**预期结果**:

- 房主创建房间成功，获得唯一邀请码。
- 其他玩家凭有效邀请码加入成功。
- 重复占座请求失败，返回 `seat_taken`。

## 场景 4：多人轮流完成一手正式牌局

1. 在房间中确认达到最少开局人数。
2. 由房主发起开局。
3. 观察系统创建当前正式牌局。
4. 轮到某位玩家时，由该玩家账号提交动作。
5. 让未轮到的另一位玩家尝试提交动作。
6. 持续推进直到该手牌通过弃牌结束或进入摊牌结算。

**预期结果**:

- 只有房主可以执行开局操作。
- 只有当前行动玩家可以提交动作。
- 非当前行动玩家请求被拒绝，返回 `not_your_turn` 或 `forbidden`。
- 一手牌完成后生成结果记录并更新所有相关玩家余额。

## 场景 5：查看战绩与一致性

1. 进入房间最近牌局结果列表。
2. 检查刚完成手牌的赢家、参与者和输赢金币。
3. 分别使用 `owner01`、`player02`、`player03` 查看个人战绩页。
4. 对照对应玩家的钱包余额与最近流水。

**预期结果**:

- 房间历史能展示刚完成的每手牌结果。
- 用户个人战绩与房间历史在同一手牌上数据一致。
- 钱包余额和牌局输赢金额能够互相解释。

## 场景 6：独立规则测试/回放页面

1. 打开现有规则引擎测试页面。
2. 创建一手随机测试牌局。
3. 执行若干动作并保存历史。
4. 进行只读回放。
5. 如实现支持，尝试调试设牌。

**预期结果**:

- 测试页仍可独立工作。
- 回放仍返回只读快照。
- 测试页能力不会要求用户先进入正式房间。
- 测试页调试设牌不会进入正式多人房间语义。

## MySQL 与 SQLite 验收差异说明

### MySQL 模式

- 作为正式多人产品默认运行模式
- 重点关注唯一约束、并发更新和事务一致性
- 建议重点验收注册并发、昵称冲突、房间邀请码唯一性和结算事务原子性

### SQLite 模式

- 作为本地开发、规则回归和轻量测试模式
- 重点关注文档语义一致性，而非生产级并发能力
- 应验证接口行为、错误码、页面流程和结果数据结构与 MySQL 模式保持一致
