# 研究记录：多人德州扑克对战 v1

## 决策 1：生产默认使用 MySQL，开发/测试允许 SQLite

- **Decision**: 生产环境默认使用 MySQL；本地开发和自动化测试允许切换为 SQLite。
- **Rationale**: 本版本已经引入账号、房间、钱包、每手牌结果和多用户并发写入语义。MySQL 更适合处理唯一索引、事务、并发更新和后续扩展。SQLite 仍保留其低运维成本优势，适合作为本地演示、规则回归和快速测试存储。
- **Alternatives considered**:
  - 仅使用 SQLite：部署简单，但对正式多人产品的并发模型和未来扩展不够稳妥。
  - 仅使用 MySQL：可以统一环境，但会提高本地开发和轻量测试门槛。
- **Trade-off**: 双模式要求领域模型和契约不能依赖某个数据库方言特性，需要在文档中明确“运行配置差异，不是两套业务逻辑”。

## 决策 2：多人首版采用“多真实账号 + 房间内轮流操作”，不做实时推送

- **Decision**: 首版多人交互模式为房主建房、邀请码加入、多真实账号轮流操作；不引入 WebSocket 实时广播。
- **Rationale**: 当前仓库已有稳定的单机规则引擎与测试回放能力，但尚未具备正式账户体系和房间领域。先做“谁可以操作、何时可以操作、结果如何归档”能最直接完成产品化闭环，同时控制技术复杂度。
- **Alternatives considered**:
  - 直接做实时联机：更像成熟游戏产品，但需要连接管理、广播、重连、状态同步和时序一致性治理，超出 v1 文档范围。
  - 假多人或脚本陪玩：实现更省，但不满足“真实多人对战”的目标。
- **Trade-off**: 首版实时感会弱于在线游戏，但换来更清晰的接口边界和可控的实现风险。

## 决策 3：登录方式选用用户名 + 密码，昵称与登录标识分离

- **Decision**: 用户以用户名 + 密码登录；昵称独立存储，作为全站唯一展示名，可在资料页修改。
- **Rationale**: 用户名适合作为稳定登录标识，昵称则用于牌桌和战绩展示。将两者拆分可避免“改昵称即改登录标识”的复杂耦合。
- **Alternatives considered**:
  - 邮箱登录：长期更利于找回密码，但当前版本不接消息通道，收益有限。
  - 手机号登录：会引入更强的格式和短信能力依赖。
  - 仅用户名且昵称与其同字段：会让展示名修改能力与登录体系耦合。
- **Trade-off**: 需要同时维护两个唯一字段，但能换来更清晰的产品语义。

## 决策 4：充值系统为“确认后直接模拟成功”，余额累加并记录流水

- **Decision**: 提供固定充值档位；用户点击任意档位后先提示，再直接模拟充值成功，钱包余额累加对应金额，并生成流水。
- **Rationale**: 该版本明确不接真实支付，因此充值系统的核心价值不是真支付链路，而是建立钱包与后续牌局结算的稳定语义。
- **Alternatives considered**:
  - 余额重置到档位金额：便于测试，但不符合“充值”的用户直觉。
  - 接入沙箱支付：仍会显著放大范围和状态机复杂度。
- **Trade-off**: 需要在文档和接口中明确这是模拟成功，而非第三方支付状态。

## 决策 5：正式多人房间复用规则引擎，但调试发牌仅留在测试页

- **Decision**: 正式多人牌局复用已有规则引擎能力；调试指定牌、只读回放和单机测试语义保留在独立规则测试页，不作为正式房间入口能力。
- **Rationale**: 现有项目的规则测试页具有高回归价值。如果直接把调试能力暴露到正式多人房间，会混淆测试能力与产品能力的边界。
- **Alternatives considered**:
  - 正式房间也允许调试发牌：便于内部验收，但会破坏正式对战语义。
  - 废弃测试页：会损失当前规则验证资产。
- **Trade-off**: 系统需要在文档和契约上维持两套路径：正式多人流程与测试流程。

## 决策 6：每手牌结果独立落库，并与钱包变更原子一致

- **Decision**: 每手牌结束时，必须以单手牌为粒度保存结果明细，同时以原子事务更新相关用户的钱包余额和钱包流水。
- **Rationale**: 多人房间产品中，战绩与余额是一体两面。若手牌结果和钱包更新可能部分成功，用户将无法信任系统账面结果。
- **Alternatives considered**:
  - 只保存整局汇总：会丢失单手牌可追溯性。
  - 异步写战绩：风险是余额和战绩暂时不一致，增加补偿复杂度。
- **Trade-off**: 事务一致性要求更高，但这是正式产品可信度的必要代价。

## 风险与后续扩展位

### WebSocket 实时化

- 风险：引入连接生命周期、广播顺序、断线重连和多终端状态一致性。
- 扩展位：当前接口可先以轮询或手动刷新方式设计，后续再增加事件推送层。

### 大厅匹配

- 风险：会新增公开房间发现、匹配策略和取消匹配状态流转。
- 扩展位：现有房间领域可在未来增加 `visibility` 或 `matchmaking` 类型。

### 真实支付接入

- 风险：需要支付订单、回调验签、失败重试、退款和审计。
- 扩展位：当前充值档位和钱包流水结构可为未来真实支付预留 `provider`、`orderNo`、`externalStatus` 字段。

### 托管与超时机制

- 风险：会把“非当前玩家不能操作”的简单鉴权，升级为带倒计时和自动动作的状态机。
- 扩展位：当前文档先不定义超时行为，只保留后续在 `HandParticipant` 或房间配置里扩展的空间。
